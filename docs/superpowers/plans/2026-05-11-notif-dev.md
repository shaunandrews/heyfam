# HeyFam Notification Dev & Test Setup Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Make the three notification channels (email, web push, SMS) testable end-to-end on the local Studio site without consuming Twilio/Postmark credit and without "we'll just check this in staging" as the answer. Email lands in **Mailpit** at `localhost:8025`. Web push works against a real Chrome on `localhost` after a one-time `npx web-push generate-vapid-keys`. SMS sends are intentionally short-circuited to `error_log` when no Twilio creds are present (today the calls fail silently — we fix it to log the body before the call). A `wp heyfam send-test-notification` CLI lets us dispatch a one-off through `FanOut::dispatch()` so we can iterate on copy and rendering without staging a real content event.

**Architecture:** No new runtime dependencies. The send pipeline (`FanOut → Push/Email/SMS`) is unchanged. We:

1. Run a single-binary Mailpit listening on `127.0.0.1:1025`. We point WP at it by adding `phpmailer_init` to a tiny "dev mailer" loaded only when an env flag is set (`HEYFAM_DEV_MAIL=mailpit`). No SMTP plugin needed in WordPress — we hijack the existing `PHPMailer` instance.
2. Document a one-time `npx web-push generate-vapid-keys --json` step. The keys get pasted into `wp-config.php` via `putenv()` (the existing pattern). On `localhost`, browsers happily register service workers and accept push subscriptions; the FCM/Apple/Mozilla push endpoints don't care about the origin's TLS or hostname — they only care that the VAPID JWT matches the public key in the subscription. So a real push from local PHP to the user's local Chrome via the real push service works.
3. Add a `wp heyfam send-test-notification` WP-CLI command that resolves a user + kind, builds a fake payload, and calls `FanOut::dispatch()` directly. This is the cheap "tweak copy without staging a real event" loop the orchestrator asks for.
4. Add a proper dev short-circuit in `Notifs\SMS::send()` so when Twilio creds are absent, the SMS body is written to `error_log` (with a clearly tagged prefix) instead of throwing inside the Twilio SDK. This formalizes the behavior that the user assumed already existed.

The `wp heyfam …` CLI namespace is co-owned with the **Test Infrastructure & Dev Helpers** workstream. We register *this* command in `plugins/heyfam-core/src/Cli/SendTestNotificationCommand.php` and the *bootstrap* of `wp heyfam` in `plugins/heyfam-core/src/Cli/Bootstrap.php`. The test-infra workstream adds its commands (`reset-me`, `seed-demo`) into the same `Cli/` directory and registers them in the same bootstrap. We define the loader so the two plans don't collide. (Directory casing is `Cli/` to match WordPress's PascalCase convention used elsewhere in this repo — `HeyFam\Core\Notifs`, `HeyFam\Core\Auth`, `HeyFam\Core\Fams`.)

**Tech Stack:** PHP 8.1+, WordPress 6.5+ multisite, Studio (local), Mailpit (single binary, optional Homebrew tap), `npx web-push` (no global install), WP-CLI (Studio ships with it).

**Reference docs:** [Mailpit install](https://mailpit.axllent.org/docs/install/), [web-push library](https://github.com/web-push-libs/web-push), HeyFam v1 spec at `docs/superpowers/specs/2026-05-08-heyfam-v1-design.md`.

**Depends on:** None. Runs in **Phase A** in parallel with the **Test Infrastructure** plan. This plan owns the creation of `plugins/heyfam-core/src/Cli/Bootstrap.php`; test-infra modifies it to add `reset-me` and `seed-demo`. Land this plan first within Phase A.

## Open Questions Resolved

| Question | Decision | Why |
|---|---|---|
| Mailpit *inside* Studio container, or alongside via Docker? | **Alongside, via Homebrew (`brew install mailpit && brew services start mailpit`) or the single static binary.** | Studio is opaque — no documented hook to run sidecar processes. Mailpit binds `127.0.0.1:1025` and `127.0.0.1:8025` on the host; Studio's PHP can reach `127.0.0.1:1025` because Studio runs on the host network (it's a macOS app, not a container). Verified by the fact that Studio sites can already `getenv()` and read host filesystem via symlinks. |
| One-time `npx web-push generate-vapid-keys`? | **Yes — documented, with output pasted directly into `wp-config.php` via `putenv()` calls.** Same shape as Twilio creds in `.env.local.example`. | The keys must be stable per-environment (changing them invalidates every existing subscription). Generating once and storing in the per-site `wp-config.php` mirrors how the staging plan handles secrets. |
| Does push work from `localhost` to a real browser? | **Yes.** | VAPID push endpoints (FCM, Mozilla, Apple) are public services. The subscription's `endpoint` URL goes from the browser back to the server, and the server posts to that URL with a JWT signed by the VAPID private key. Neither the origin's hostname nor TLS affects this — Chrome registers service workers on `localhost` without HTTPS by design. The only friction is the user must grant per-origin notification permission (one prompt). |
| Where does the SMS dev log message go? | **`error_log()` with `[heyfam dev SMS]` prefix.** | Matches the existing `[heyfam dev]` Verify-code log convention in `PhoneSignup::start_verification()`. Studio routes `error_log` to the site's `wp-content/debug.log` when `WP_DEBUG_LOG=true`. |
| Should the dev mailer log even when Mailpit is the mode? | **Yes — additionally log a one-line summary to `error_log` (subject + recipient).** | Lets us notice fan-out misroutes even without flipping over to Mailpit's UI. Cheap to leave on. |
| Should `send-test-notification` actually deliver, or just dry-run? | **Real send through `FanOut::dispatch()`.** Add `--dry-run` flag that returns the payload without sending. | Real send is the whole point: catch silent breakage in copy or push payload size. `--dry-run` is the safety net for sketchy targets. |
| `wp heyfam` namespace — who owns the bootstrap? | **This plan creates `Cli/Bootstrap.php`. Test-Infrastructure plan adds its commands by *modifying* that file (single new line per command).** | Avoids two plans both creating the same file. The Bootstrap is intentionally trivial — a `WP_CLI::add_command()` per command, alphabetized. Pascal `Cli/` matches the repo's `HeyFam\Core\Notifs`, `HeyFam\Core\Auth` convention. |

### Step 3: Create `Notifs/DevMailer.php`

```php
<?php
namespace HeyFam\Core\Notifs;

/**
 * Re-routes wp_mail through Mailpit (or a logfile) when a dev env flag is set.
 *
 * Modes (HEYFAM_DEV_MAIL):
 *   "mailpit" — send to 127.0.0.1:1025 (no auth). View at http://localhost:8025.
 *   "log"     — error_log() a one-line "[heyfam dev mail] To=... Subject=..." plus body.
 *   ""        — no-op (production / staging path).
 *
 * Why hook PHPMailer directly instead of using a plugin like WP Mail SMTP:
 *  - Zero extra dependencies on the local site.
 *  - Works identically whether you toggle by env var or by hand in wp-config.
 *  - The "log" fallback means a contributor can verify our fan-out without
 *    installing Mailpit at all.
 */
final class DevMailer {
    public function __construct() {
        $mode = getenv( 'HEYFAM_DEV_MAIL' ) ?: '';
        if ( $mode === '' ) return;
        if ( $mode === 'mailpit' ) {
            add_action( 'phpmailer_init', [ $this, 'route_to_mailpit' ] );
        }
        // Always also log a summary so debug.log shows what we tried to send.
        add_action( 'phpmailer_init', [ $this, 'log_summary' ] );
    }

    public function route_to_mailpit( \PHPMailer\PHPMailer\PHPMailer $mailer ): void {
        $mailer->isSMTP();
        $mailer->Host       = '127.0.0.1';
        $mailer->Port       = 1025;
        $mailer->SMTPAuth   = false;
        $mailer->SMTPAutoTLS = false;
        $mailer->SMTPSecure = '';
        // Mailpit accepts any From; keep the wp_mail-provided value.
    }

    public function log_summary( \PHPMailer\PHPMailer\PHPMailer $mailer ): void {
        $to = is_array( $mailer->getToAddresses() )
            ? implode( ',', array_map( static fn( $a ) => $a[0] ?? '', $mailer->getToAddresses() ) )
            : '';
        $mode = getenv( 'HEYFAM_DEV_MAIL' ) ?: '';
        error_log( sprintf(
            '[heyfam dev mail %s] To=%s Subject=%s',
            $mode,
            $to,
            $mailer->Subject
        ) );
        if ( $mode === 'log' ) {
            // Print the body too in pure-log mode.
            error_log( '[heyfam dev mail body] ' . substr( wp_strip_all_tags( $mailer->Body ), 0, 800 ) );
            // Short-circuit the send by leaving the mailer in SMTP mode pointing at a
            // closed port. Cleaner: throw, but PHPMailer will surface that to wp_mail
            // as a failure and our Email::send() returns false, which is exactly what
            // we want — the queue does not retry.
            $mailer->isSMTP();
            $mailer->Host = '127.0.0.1';
            $mailer->Port = 65535; // guaranteed-closed sentinel; send fails silently
        }
    }
}
```

### Step 5: Document subtleties

We do not commit any code here. Note in the doc (Task 5):

- Chrome treats `http://localhost` (and `*.localhost`) as a "secure context" for the push/Notification APIs. Other hostnames like `heyfam.test` require HTTPS. **If your Studio URL is `heyfam.test`** (the default), either:
  - (a) add `http://heyfam.test` to `chrome://flags/#unsafely-treat-insecure-origin-as-secure`, restart Chrome; or
  - (b) point your hosts file or Studio config so the site is served from `http://localhost:NNNN`. Studio supports this — check Studio settings → "Custom port".
- Safari desktop does not implement Web Push API the same way. For Safari push testing, use the iOS PWA path (16.4+, "Add to Home Screen").
- If you change VAPID keys, every existing subscription becomes invalid (server gets `403` from push service). Delete the rows from `wp_heyfam_push_subscriptions` and re-subscribe.

Skip. Task 5 commits the documentation that summarizes everything in this task.

### Step 2: Decide how to call `FanOut::dispatch()`

`FanOut::dispatch()` is currently `private`. Two options:

- **(a)** Bump it to `public` (lowest friction, minor encapsulation cost). One-line diff.
- **(b)** Add a thin public wrapper `FanOut::dispatch_for_test( int $user_id, int $blog_id, string $event, array $payload ): void` that just calls the private method. Cleaner separation.

Pick **(a)** — visibility change in `Notifs/FanOut.php` from `private function dispatch` to `public function dispatch`. It's already called only from within `FanOut`'s own methods; making it public costs us nothing semantically. The CLI is a legitimate caller.

(If you'd rather not touch `FanOut`, do (b) and add the wrapper instead. Pick one and stick to it.)

### Step 3: Create `Cli/SendTestNotificationCommand.php`

```php
<?php
namespace HeyFam\Core\Cli;

use HeyFam\Core\Auth\Authorization;
use HeyFam\Core\Notifs\FanOut;

/**
 * Send a one-off test notification to a real user, through the real fan-out
 * pipeline, so we can verify push/email rendering without staging a content event.
 *
 * ## OPTIONS
 *
 * --user=<id>
 * : User ID to deliver to (the recipient — *not* the sender). Required.
 *
 * [--blog=<id>]
 * : Fam (blog) ID for the notification. Defaults to the current site.
 *
 * [--kind=<kind>]
 * : One of: post, comment, reaction. Default: post.
 *
 * [--target=<post_id_or_comment_id>]
 * : The post or comment id to reference. Default: most recent post on the fam.
 *
 * [--dry-run]
 * : Print the payload that would be dispatched, but do not actually send.
 *
 * ## EXAMPLES
 *
 *     # Fire a "new post" notification to user 2 on the current fam
 *     wp heyfam send-test-notification --user=2 --kind=post
 *
 *     # Comment notification, targeting a specific comment
 *     wp heyfam send-test-notification --user=2 --kind=comment --target=14
 *
 *     # Just show the payload — don't actually push/email/SMS
 *     wp heyfam send-test-notification --user=2 --kind=reaction --dry-run
 */
final class SendTestNotificationCommand {
    public function __invoke( array $args, array $assoc ): void {
        $user_id = (int) ( $assoc['user'] ?? 0 );
        $blog_id = (int) ( $assoc['blog'] ?? get_current_blog_id() );
        $kind    = (string) ( $assoc['kind'] ?? 'post' );
        $target  = isset( $assoc['target'] ) ? (int) $assoc['target'] : 0;
        $dry     = ! empty( $assoc['dry-run'] );

        if ( ! $user_id || ! get_userdata( $user_id ) ) {
            \WP_CLI::error( "User $user_id not found." );
        }
        if ( ! in_array( $kind, [ 'post', 'comment', 'reaction' ], true ) ) {
            \WP_CLI::error( "Unknown --kind: $kind (use post|comment|reaction)" );
        }

        $payload = Authorization::in_blog( $blog_id, function () use ( $kind, $target ) {
            return $this->build_payload( $kind, $target );
        } );
        if ( is_wp_error( $payload ) ) {
            \WP_CLI::error( $payload->get_error_message() );
        }

        \WP_CLI::log( "→ kind=$kind user=$user_id blog=$blog_id" );
        \WP_CLI::log( '  title: ' . $payload['title'] );
        \WP_CLI::log( '  body:  ' . $payload['body'] );
        \WP_CLI::log( '  url:   ' . $payload['url'] );
        if ( $payload['sms'] ) \WP_CLI::log( '  sms:   ' . $payload['sms'] );

        if ( $dry ) {
            \WP_CLI::success( 'Dry run; nothing sent.' );
            return;
        }

        // Map kind → event used by Prefs::should_notify (matches FanOut keys).
        $event = match ( $kind ) {
            'post'     => 'posts',
            'comment'  => 'comments',
            'reaction' => 'reactions',
        };

        ( new FanOut() )->dispatch( $user_id, $blog_id, $event, $payload );
        \WP_CLI::success( "Dispatched $kind notification to user $user_id." );
    }

    private function build_payload( string $kind, int $target ): array {
        $blog = get_blog_details( get_current_blog_id() );
        if ( $kind === 'post' ) {
            $post = $target ? get_post( $target ) : ( get_posts( [ 'numberposts' => 1 ] )[0] ?? null );
            if ( ! $post ) return [
                'title' => "Test post in $blog->blogname",
                'body'  => 'A pretend post fired from `wp heyfam send-test-notification`.',
                'url'   => $blog->siteurl . '/',
                'sms'   => "Test ($blog->blogname): pretend post — $blog->siteurl/",
                'tag'   => 'test-post-' . time(),
            ];
            return [
                'title' => sprintf( '%s in %s', get_userdata( $post->post_author )->display_name, $blog->blogname ),
                'body'  => wp_trim_words( $post->post_content, 18 ),
                'url'   => $blog->siteurl . '/p/' . $post->ID,
                'sms'   => sprintf( '%s (%s): %s — %s/p/%d',
                    get_userdata( $post->post_author )->display_name, $blog->blogname,
                    wp_trim_words( $post->post_content, 12 ),
                    $blog->siteurl, $post->ID
                ),
                'tag'   => 'test-post-' . $post->ID,
            ];
        }
        if ( $kind === 'comment' ) {
            $comment = $target ? get_comment( $target ) : null;
            $post_id = $comment ? (int) $comment->comment_post_ID : ( get_posts( [ 'numberposts' => 1, 'fields' => 'ids' ] )[0] ?? 0 );
            return [
                'title' => sprintf( 'Test comment in %s', $blog->blogname ),
                'body'  => $comment ? wp_trim_words( $comment->comment_content, 18 ) : 'Pretend comment body',
                'url'   => $blog->siteurl . '/p/' . $post_id . ( $comment ? '#c-' . $comment->comment_ID : '' ),
                'sms'   => '',
                'tag'   => 'test-comment-' . ( $comment ? $comment->comment_ID : time() ),
            ];
        }
        // reaction
        return [
            'title' => 'Test reaction in ' . $blog->blogname,
            'body'  => 'Someone reacted to your ' . ( $target ? 'post' : 'something' ) . '.',
            'url'   => $blog->siteurl . '/',
            'sms'   => '',
```

## Task 4: SMS — proper dev log mode

**Files:**
- Modify: `plugins/heyfam-core/src/Notifs/SMS.php`

### Step 1: Add a dev short-circuit to `SMS::send()`

The user's brief implied a dev bypass already exists; verify shows only `PhoneSignup` has one. The actual production-path call to `TwilioClient::rest()->messages->create()` with empty creds throws inside the SDK, gets caught, and is `error_log`ged as a failure — i.e., the SMS is silently dropped on the floor of "[heyfam] SMS send failed: …". This is fragile: the failure path is *exception flow* through the Twilio SDK, which is slow and noisy in logs.

Replace `send()` with:

```php
public static function send( string $to_e164, string $body ): bool {
    if ( self::is_opted_out( $to_e164 ) ) return false;

    // Dev mode: no Twilio creds. Log the body and short-circuit. Same condition
    // the verify flow already uses (Auth\PhoneSignup::is_dev_mode).
    if ( ! getenv( 'TWILIO_ACCOUNT_SID' ) ) {
        error_log( sprintf( '[heyfam dev SMS] To=%s Body=%s', $to_e164, $body ) );
        return true;
    }

    try {
        TwilioClient::rest()->messages->create( $to_e164, [
            'messagingServiceSid' => TwilioClient::messaging_sid(),
            'body'                => $body,
        ] );
        return true;
    } catch ( TwilioException $e ) {
        error_log( '[heyfam] SMS send failed: ' . $e->getMessage() );
        return false;
    }
}
```

Two notes:

1. We return `true` in dev mode. The caller (`FanOut::dispatch()`) discards the boolean today, so the value doesn't matter for behavior — but `true` reads as "delivered to the dev-log sink, as intended."
2. The dev check is `! getenv('TWILIO_ACCOUNT_SID')`, identical to `PhoneSignup::is_dev_mode()`. Could DRY into a shared helper, but two call sites isn't worth the indirection.

```bash

tail -20 wp-content/debug.log | grep 'heyfam dev SMS'
# expect: [heyfam dev SMS] To=+15555550100 Body=...
```

If user 2 has no `phone_e164` meta, no SMS is attempted; that's fine — `FanOut::dispatch()` checks for the phone before calling `SMS::send`.

|---|---|---|
| No notification permission prompt | Browser blocks default prompt on first load | DevTools console: `await Notification.requestPermission()` |
